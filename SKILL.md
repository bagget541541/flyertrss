# 文档输出验收（doc-qa-reviewer）

## 任务目标

对输入文档做**五类质量检查**，输出 Markdown 验收报告。每条问题须包含：位置（章节/段落关键词）、问题描述、修改建议。

**输入**：`.docx` 或 `.doc` 文档  
**输出**：`qa_report_MMDD.md`（Markdown）

---

## 工作流

### Step 1：读取文档

```bash
# 优先用 extract-text
extract-text input.docx

# .doc 须先转换
python scripts/office/soffice.py --headless --convert-to docx input.doc
```

提取全文后，**整体通读一遍**，建立对文档结构、主题、逻辑主线的整体印象，再逐项检查。

---

### Step 1.5：判断文档类型（决定是否做 E 类检查）

通读后，判断文档的**分发对象**：

| 类型 | 判断依据 | E 类检查 |
|------|---------|---------|
| **对外文档** | 公众号文章、产品说明、用户手册、销售材料、面向客户的方案介绍 | ✅ 执行 |
| **混合型** | 内部用但写法接近对外（如信用卡资讯汇总、面向非技术干系人的汇报） | ✅ 执行，说明判断依据 |
| **纯内部技术文档** | 详设、接口协议、架构方案、标书技术部分、内部测试报告 | ❌ 跳过，报告中记一行说明 |

判断结果写入报告头部"文档类型"字段，作为 E 类检查是否出现的依据。

---

### Step 2：五类检查

#### 检查 A — 标题与格式一致性

扫描所有标题（一级、二级、三级），检查：

| 检查项 | 说明 |
|--------|------|
| 层级编号格式 | 同级标题编号风格是否统一（如"一、"vs"1."混用） |
| 字体/加粗规律 | 同级标题加粗、字号、括号风格是否一致 |
| 标题末尾标点 | 标题后不应有句号，是否有漏网之鱼 |
| 平行结构 | 同级标题词性/句式应平行（如"功能概述"vs"对系统进行部署"） |
| 缺失标题 | 目录中有但正文找不到，或正文有但目录没有 |

**标记格式**：`[格式-A01]` 位置 → 问题 → 建议

---

#### 检查 B — 图文匹配

扫描所有图片引用（"如图X所示"/"见下图"/"图X-Y"等）及图题，检查：

| 检查项 | 说明 |
|--------|------|
| 引用与图位置 | 文中引用图编号，对应位置是否真的有该图 |
| 图题缺失/多余 | 有图无题，或有题无图 |
| 图题与内容匹配 | 图题描述与图实际内容是否对应 |
| 图编号连续性 | 图1、图2…是否连续，有无跳号或重号 |
| 文字描述与图矛盾 | 正文说"三个模块"但图里画了四个 |

**若文档无图**：记录"文档未含图片，跳过图文检查"。

**标记格式**：`[图文-B01]` 位置 → 问题 → 建议

---

#### 检查 C — 逻辑一致性

通读全文，检查前后逻辑：

| 检查项 | 说明 |
|--------|------|
| 术语统一 | 同一概念前后名称不同（如"总控节点"vs"主控单元"） |
| 数字/参数前后矛盾 | A章说"4路输入"，B章变成"6路" |
| 结论与正文脱节 | 结论/建议章节的观点在正文中找不到依据 |
| 章节间衔接 | 上一节说"详见第X节"，第X节内容是否真的对应 |
| 时态/状态矛盾 | "已完成"vs"待规划"描述同一事项 |

**标记格式**：`[逻辑-C01]` 位置A + 位置B → 矛盾内容 → 建议

---

#### 检查 D — 废话识别

逐段检查信息密度，标记低价值内容：

| 废话类型 | 示例 |
|---------|------|
| 空洞背景铺垫 | "随着信息化时代的到来，越来越多的……" |
| 重复前文内容 | 摘要/前言与正文完全同质，无增量信息 |
| 过度解释显而易见的内容 | "计算机是一种电子设备，可以进行计算" |
| 无实质内容的过渡句 | "综上所述，本章对上述内容进行了介绍" |
| 罗列不带结论的清单 | 列了7条但没说选哪个或为什么 |

每处废话给出：删除建议 或 压缩建议（保留核心信息的精简版）。

**标记格式**：`[废话-D01]` 位置 → 废话类型 → 建议（删/改）

---

#### 检查 E — 读者视角（仅对外/混合型文档）

以**读者**而非作者身份重读全文，检查：

| 问题类型 | 说明 | 示例 |
|---------|------|------|
| 主语错位 | 以产品/系统/我方为主语，而非读者 | "本系统支持X" → "你可以用X来做Y" |
| 缺"对我有什么用" | 罗列功能/权益，未说明解决什么问题或带来什么收益 | 列了5条权益，没说哪条最值得关注 |
| 术语未解释 | 使用内部缩写、行业黑话未加说明 | "DHT""HIL"直接出现无注释 |
| 行动指引模糊 | 读者看完不知道该做什么、去哪里、截止何时 | "可参考相关活动页面"而非"报名路径：掌银APP→X→Y" |
| 负面信息回避 | 限制条件、门槛、风险藏在不显眼位置，正面信息过度放大 | 标题写"月返100元"，正文小字才说"需月消费3000元" |
| 信息优先级失序 | 读者最关心的内容排在后面，次要内容占据前置位置 | 先写申请流程，后写"是否值得申请" |

**标记格式**：`[视角-E01]` 位置 → 问题类型 → 建议

**注意**：E 类是程度判断，不是非黑即白。发现明显偏离读者视角的才标记，轻微措辞偏差不算。

---

### Step 3：生成报告

报告结构如下，按发现问题数量由多到少排序各类检查：

```markdown
# 文档验收报告

**文档**：[文件名]  
**验收日期**：[日期]  
**文档类型**：[对外 / 混合型 / 纯内部技术文档]（附判断依据一句话）  
**问题总数**：A类 X 条 | B类 X 条 | C类 X 条 | D类 X 条 | E类 X 条（或"跳过"）| 合计 X 条
```

## 输出规范

- 每类问题**实事求是**：找到几条写几条，不凑数，不遗漏显眼问题
- 问题定位用**原文关键词**，不用"第X页"（页码随编辑变化）
- 修改建议**具体可操作**，不写"建议完善该部分"这种废话
- 总体评估**不超过3句**，直接点明核心问题
- 报告本身不能有废话，以本 skill 检查的标准约束自身输出

---

## 文档类型适配说明

| 文档类型 | 重点检查项 |
|---------|-----------|
| 技术方案/详设 | C（逻辑）> B（图文）> A（格式）> D（废话），跳过 E |
| 标书/投标文件 | A（格式）> C（逻辑）> D（废话）> B（图文），跳过 E |
| 汇报/PPT转文档 | D（废话）> C（逻辑）> A（格式）> B（图文），跳过 E |
| 用户手册/操作指南 | B（图文）> E（视角）> C（逻辑）> A（格式）> D（废话） |
| 公众号/产品说明 | E（视角）> D（废话）> C（逻辑）> A（格式）> B（图文） |
| 混合型（内部用但对外写法） | C（逻辑）> E（视角）> D（废话）> A（格式）> B（图文） |

根据文档实际类型自动调整检查重点顺序。
