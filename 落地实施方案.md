## 二、已验证结论

### ✅ 可行
- `flyert.com.cn` 通过代理可达，HTTP 200
- 列表页公开可读，**无需登录**
- 结构标准 Discuz!，`BeautifulSoup` 可直接解析
- 每页约 23~30 条，版块共 **34 页**（约 750 条）
- 去重（基于 `tid`）和去噪规则已验证有效
- Playwright（真实 Chromium）可渲染论坛页面，桥接 JS 校验

### ⚠️ 注意点
- 必须以代理访问（后端验证 IP），**直连不可用**
- 页面编码 **GBK**，需转码
- 超过第 1 页的 URL 格式不同（`forum.php?mod=forumdisplay&fid=59&page=N`）

### 🛡️ WAF 对抗实战发现（2026-06-01 实测）

### 部署方案实测总结

2026-06-02 对以下方案进行了完整实测：

| 方案 | 实测结果 | 根因 |
|------|:--------:|------|
| **A — COS 静态网站** | ❌ 浏览器下载文件 | 2024年后新桶强制 `Content-Disposition: attachment`，无法绕过 |
| **A + CDN 加速** | ❌ 控制台找不到入口 | 新控制台移除了默认 CDN 域名开启入口，仅保留自定义域名 |
| **D — SCF Event 函数 + 函数 URL** | ❌ 下载文件 | 函数 URL 平台覆盖响应头为 `attachment` + `application/json` |
| **D — SCF Web 函数 + 函数 URL** | ❌ 仍然下载 | Web 函数也无法绕过函数 URL 的安全策略 |
| **Vercel** | ❌ 国内无法访问 | 域名被封锁 |
| **Gitee Pages** | ❌ 服务已下架 | Gitee 不再提供 Pages 服务 |
| **E — GitHub Pages** | ✅ **完全可用** | 正常渲染 `.html`，无响应头限制，国内可访问 |

**结论**：**GitHub Pages 是当前唯一可行的免费方案**。

| 探测方式 | 首页 `/` | 论坛页 `/forum-creditcard-1.html` | 结论 |
|---------|:-------:|:-------:|------|
| **curl**（单 UA + 代理） | 200 | 403 | curl 的 TLS 指纹被 WAF 识别 |
| **curl**（完整浏览器 headers + cookie jar） | 200 | 403 | 仅改 headers 无效 |
| **httpx** （Python） | 200 | 403 | 与 curl 同级，TLS 指纹特征类似 |
| **requests** | 200 | 403 | 同上 |
| **Playwright**（headless Chrome） | 200 | 403 | IP 被封时连真实浏览器也无用 |
| Playwright（同 IP 冷却后） | 200 | **应可恢复** | 等待 IP 冷却 |

**关键发现**：
1. **IP 级封禁** — 论坛路径触发 WAF 后，整条链路 IP 被加入临时黑名单（非路径级校验）
2. **冷却周期** — 密集触发（2h 内 30+ 次请求）后需 **30min~2h** 解封
3. **TLS 指纹非主要因素** — 连真实 Chrome 都被挡说明是 IP 白名单/黑名单机制
4. **WAF 检测器** — `"403 Forbidden" in html[:200] and "Access Denied" in html[:200]` 可靠

## 三、技术架构

```
flyert.com.cn
    │
    ├─ ① Playwright（真实 Chromium） ← WAF 穿透首选
    │      └─ 失败时 ↓
    ├─ ② httpx（Python，带代理）       ← 后备
    │      └─ 失败时 ↓
    └─ ③ curl（带 cookie jar + 代理）  ← 最后手段
           │
           ▼
      WAF 检测 ──→ 命中？── 冷却 30s/2min/10min 重试
           │
           ▼

           │
           │ 去重(seen_tids.json)
           │ 去噪(关键词/互动阈值)
           ▼

           │
           │ LLM API (DeepSeek) 分类
           ▼
      日报.md  ←─ Jinja2 模板渲染
```

### Step 3 ✅ LLM 分类 + 规则降级 — 基础实现已完成

**输入**：`threads_filtered.json` 中的每条帖子

**架构**：LLM 优先 → 失败或不可用时自动降级为关键词规则

| 字段 | 来源 |
|------|------|
| 银行分类 | 直接从帖子分类标签取（95% 已有） |
| 主题分类 | `权益变化` / `申卡下卡` / `活动优惠` / `疑问求助` / `用卡经验` / `额度讨论` / `其他` |
| 一句话摘要 | 标题本身已高度浓缩，多数无需额外摘要 |
| 热度标签 | 根据阅读/回复数打标 `🔥热门`（回复 > 30）|

**LLM 调用策略**：
- 对全部保留帖子做**批量**调用（一次传全部）
- 提示词要求：**仅分类，不重述标题**（省钱）
- 模型：DeepSeek / 4o-mini 均可，每日消耗 < 0.1 元
- 降级方案：6 组关键词规则，覆盖 95% 帖子（`summary.py` 中 `RULES` 字典）

## 五、风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| flyert 改版或字段变更 | 低 | BeautifulSoup 选择器报错时日志告警，手动修一次 |
| 代理失效 / 端口变化 | 低 | `PROXY_LIST` 配置多组代理自动故障转移 |
| **WAF IP 封禁** | **高**（初始阶段） | 见下文详细策略 |
| LLM API 不可用 | 低 | 降级为关键词规则分类（95% 已有分类标签） |

### 🛡️ WAF 封禁详细应对策略

| 时机 | 策略 | 说明 |
|------|------|------|
| **日常运行** | 每日 1~2 次，每次仅 1 页 | ≈ 1 个请求/天，远低于 WAF 阈值 |
| **请求间隔** | 多页时 1.0~1.9s 随机抖动 | 避免固定间隔被识别为爬虫 |
| **User-Agent** | Playwright 自带真实 Chrome UA | 不需要额外伪装 |
| **代理持久化** | curl 模式使用 `cookies.txt` 持久化会话 | 模拟有状态的浏览器访问 |
| **被 403 时** | 自动重试：等 30s → 2min → 10min | 三次重试均失败则放弃当日运行 |
| **代理故障转移** | `PROXY_LIST` 数组顺序尝试 | 自动跳过不可用代理 |
| **IP 完全被封** | 关闭代理，改用直连（兜底） | 直连 IP 可能是干净的 |

### ⚡ 今日实测复盘

当天 WAF 交锋完整记录（以下均为北京时间 UTC+8）：

| 时间 | 动作 | 结果 | 教训 |
|------|------|------|------|
| 6/1 22:00~22:30 | fetcher 测试、WAF 探测（curl/httpx/Playwright） | 约 22:00 可访问，22:10 后持续 403 ❌ | 密集请求约 10 分钟后触发 WAF IP 封禁 |
| 6/1 22:30~6/2 06:30 | 冷却等待（约 8h） | 恢复可访问 ✅ | 冷却周期约 8~10 小时 |
| 6/2 06:30~06:40 | 重新测试 fetcher + WAF 探测 | 初期 200 ✅，06:35 后 403 ❌ | 再次触发，冷却期待确认 |
| 6/2 07:00~至今 | 等待冷却，改进代码 | 待验证 | ⏳ |

**核心教训**：
1. **不要密集测试飞客论坛** — 这是最大的 WAF 触发器。日常运行仅 1 次请求/天，远不足以触发封禁
2. **Playwright 非万能** — IP 被封时连真实 Chrome 也一样被挡，只是不容易被 TLS 指纹识别
3. **冷却期约 8h+** — 两次触发经验：第一次 22:00→次日 06:30（约 8.5h）才恢复，比预期长
4. **冷却等待是唯一解** — 没有绕过 IP 封禁的捷径，自动重试 + 每日限 1 次是正确策略

## 🔍 V2Net Discover 参考分析

### 站点定位
> **[discover.v2net.xyz](https://discover.v2net.xyz/)** — "V2EX 发现引擎"，以社区质量信号为权重发现 V2EX 上值得读的内容

本质是一个 V2EX **内容发现网关**，非通用爬虫工具。和我们做飞客日报的「论坛情报」定位一致。

### 技术栈（逆向分析）

| 层次 | 技术 |
|------|------|
| 框架 | **Next.js 14+**（App Router + RSC） |
| 前端 UI | **Tailwind CSS** + Lucide icons + shadcn/ui |
| 数据流 | React Server Components 流式渲染 |
| 主题 | 亮/暗/系统，localStorage 持久化 |
| 字体 | Geist（Vercel 字体） |

### API 架构

```
GET /api/topics/hot      → { topics: [...], cached: true }   # 热门（10条，按热度分）
GET /api/topics/best     → { topics: [...], cached: true }   # 精选（50条）
GET /api/topics/latest   → { topics: [...], cached: true }   # 最新（37条）
GET /api/topics/hot?node=ideas   → 按节点过滤
```

**每条 topic 数据结构**：
```json
{
  "id": 1216940,
  "title": "xxx",
  "replies": 199,
  "author": "Heimerdinger",
  "authorAvatar": "https://cdn.v2ex.com/avatar/...",
  "node": "libido",
  "nodeName": "情感问题",
  "createdAt": 1780281934,
  "lastTouched": 1780345087,
  "heatScore": 2.009,     // 🔥 核心：综合热度分
  "aiSummary": "..."       // 仅部分帖子有 AI 摘要
}
```

### 核心设计参考

| V2Net 做法 | 对我们的启发 |
|---|---|
| **`heatScore` 综合热度** — 回复 + 阅读 + 时间衰减 + 收藏/感谢 | 日报排序可引入类似算法，替代纯回复数排序 |
| **服务端缓存**（`"cached": true`）— 减少对 V2EX API 的压力 | 飞客 WAF 敏感，服务端缓存尤其重要 |
| **AI 摘要**（仅热帖显示）— 有限资源做最大效果 | 我们的 LLM 分类也走批量+轻量路径，思路一致 |
| **节点分类导航** — 左侧栏按节点过滤 | 日报按论坛自带分类标签展示，95% 已覆盖 |
| **API 返回 JSON** → 前后端分离 | 后续可拆出 API 层给其他消费端（如微信推送） |

### 关键差异

| 维度 | V2Net Discover | 我们 |
|------|:---:|:---:|
| 数据源 | V2EX 官方 API（免费、公开、无 WAF） | flyert.com.cn Discuz!（有 WAF、IP 限制） |
| 抓取难度 | **低** — 直接调 API 即可 | **高** — 需要 Playwright 绕过 WAF |
| 内容粒度 | 帖子全文 + AI 摘要 | 仅列表页标题 + 标签（详情页过度消耗带宽） |
| 前端 | Next.js SSR | 纯 Markdown 文件，自用不展示 |

**结论：V2Net 的架构设计和热度算法值得参考，但我们的主要矛盾在 WAF 绕过，而非前端/API。**

#### Step A1 — 创建 COS Bucket

3. 创建完成后，进入 **存储桶配置 > 基础配置**，开启 **静态网站**：
   - 索引文档：`index.html`

#### Step B3 — 写云函数入口 `index.py`

在项目根目录创建 SCF 入口文件：

```python
# index.py — 腾讯云 SCF 入口
import sys, os, json
from pathlib import Path

# 确保项目根目录在路径中
sys.path.insert(0, os.path.dirname(os.path.abspath(__file__)))

def main_handler(event, context):
    """SCF 定时触发器入口"""
    from cos_python_sdk_v5 import CosConfig, CosS3Client

    # 1. 运行抓取+分类
    os.chdir(Path(__file__).parent)
    import runpy
    runpy.run_path("run.py", run_name="__main__")

    # 2. 上传 _site/ 到 COS
    cfg = {
        "bucket": os.environ.get("COS_BUCKET", "flyertrss-1300000000"),
        "region": os.environ.get("COS_REGION", "ap-guangzhou"),
        "secret_id": os.environ["COS_SECRET_ID"],
        "secret_key": os.environ["COS_SECRET_KEY"],
    }
    config = CosConfig(Region=cfg["region"],
                       SecretId=cfg["secret_id"],
                       SecretKey=cfg["secret_key"])
    client = CosS3Client(config)

    src = Path("_site")
    for f in sorted(src.rglob("*")):
        if f.is_file():
            key = str(f.relative_to(src)).replace("\\", "/")
            client.upload_file(
                Bucket=cfg["bucket"],
                LocalFilePath=str(f),
                Key=key,
                EnableMD5=True,
            )
    return {"code": 0, "message": f"部署完成"}
```

> ⚠️ **注意**：SCF 环境没有真实 Chromium/Playwright，因此 fetcher 中 Playwright 模式会失败。建议在 SCF 部署前，将 `fetcher.py` 的首选策略改为 `httpx`（纯 Python HTTP 请求）。飞客的 WAF 在 SCF 腾讯云 IP 段上是否被封需单独测试。

### 方案 D：SCF 云函数代理（替代 COS 静态网站，解决下载问题）

如果代码托管在 GitHub，可以配置 Actions 自动部署到 COS，替代 `deploy_cos.py` 手动上传。

**原理**：代码推送到 `main` 分支 → GitHub Actions 运行 `run.py` 生成 `_site/` → 使用 `tencent-cos-action` 上传到 COS

新建 `.github/workflows/deploy-cos.yml`：

```yaml
# .github/workflows/deploy-cos.yml
name: Deploy to Tencent COS

on:
  push:
    branches: [main]
  schedule:
    - cron: '0 0,12 * * *'  # UTC 0:00 / 12:00 → 北京 8:00 / 20:00

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.9'

      - name: Install dependencies
        run: |
          pip install httpx beautifulsoup4 jinja2  # Playwright 在 Action 中不可用，回退 httpx

      - name: Generate daily report
        run: python run.py

      - name: Deploy to COS
        uses: tencent-oss/cos-action@v1
        with:
          secret_id: ${{ secrets.TENCENT_SECRET_ID }}
          secret_key: ${{ secrets.TENCENT_SECRET_KEY }}
          bucket: ${{ secrets.TENCENT_COS_BUCKET }}
          region: ap-guangzhou
          local_path: _site
          remote_path: /
```

> ⚠️ 注意：GitHub Actions 环境无法运行 Playwright（无桌面环境），因此 Action 中 fetcher 需自动降级为 httpx 模式。建议在 `fetcher.py` 中检测 `CI` 或 `GITHUB_ACTIONS` 环境变量来自动切换爬取策略。
